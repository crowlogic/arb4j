# arb4j axioms

Invariants of the compiler. Each axiom is non-negotiable. Violations are bugs.

## A1. `Node.generate(MethodVisitor, Class<?>)` does not mutate state

Every node class's `generate(MethodVisitor mv, Class<?> resultType)` method has exactly one effect: writing bytecode to `mv`. A node's `generate` does not:

- mutate `referencedFunctions`, `referencedVariables`, `declaredVariables`, `intermediateVariables`, `declaredIntermediateVariables`, or any other field of any `Expression`,
- register, deregister, or rename anything in any `Context`,
- invoke `compile()`, `optimize()`, `parse()`, or `resolve()` on any expression, including the owning expression, nested mappings, sibling functionals, or upstream expressions,
- construct, mutate, or splice `Node` instances into any AST,
- alter `FunctionMapping.expression`, `FunctionMapping.instance`, or any field of any mapping,
- write to `compileInProgress`, `optimized`, `variablesDeclared`, or any other compilation-phase flag,
- call any helper that performs any of the above.

Every fact a node's `generate` needs — its operand types, its referenced names, its field descriptors, its slot assignments, its initializer registration — must already be final at the moment `Expression.generate()` reaches `rootNode.generate(mv, coDomainType)`. If a fact is not established by then, the bug is at the parse/resolve/optimize phase that should have established it, not at the emission site.

The per-`Node` `generate` is read-only on the AST and on every `Expression` reachable from it. The only objects it writes to are the `MethodVisitor` (and, transitively, the `ClassVisitor` only insofar as helpers it invokes write to it).

## A2. Referenced variables are context variables only

`Expression.referencedVariables` is the set of context-variable references that require field declaration and injection. It is not a general list of every variable name occurring in an expression.

Independent variables, bound variables, summation/product indices, placeholder variables, and upstream input variables must never be entered into `referencedVariables`. They are always semantically referenced by their binding form and are propagated, when necessary, by input/bound-variable mechanisms, not by context-variable injection.

In particular:

- A variable being named in a parent, ancestor, or enclosing expression does not make it a context variable.
- A bound/index variable such as `n`, `j`, or `k` in `sum`, `prod`, `pFq`, or generated operand code must not be injected as a field through `referencedVariables`.
- A field may only be written on a generated target if the target's expression actually owns that field by the correct mechanism.
- Parent field existence does not imply child field existence.
- Child field existence implies the parent or enclosing generator path must be able to supply it by the correct mechanism.

Violating this axiom creates invalid generated bytecode such as writing `this.F.n = this.n` when helper class `F` has no `n` field, or silently drops required context variables such as `μ`, `α`, or `x`.

## A3. Recursive self fields are distinct generated instances, not `this`

A generated self-reference field such as `a.a` in class `a` must not be assigned to `this`.

Same class does not imply same instance. A field of type `a` inside class `a` means this generated object has a reference to another generated `a` instance. It does not mean the field aliases the currently evaluating object.

Recursive generated evaluators carry mutable scratch registers, cache state, initialization state, and temporary ARB objects. Assigning `this.a = this` makes recursive calls re-enter the same mutable evaluator and corrupts that state. The correct model is a separate generated instance of the same class whose context variables and referenced functions are wired correctly.

Therefore the compiler must preserve separate-instance recursive fields while ensuring every such instance receives the context fields and peer references it actually declares and uses.
